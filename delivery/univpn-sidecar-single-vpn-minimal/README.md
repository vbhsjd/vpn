# UniVPN Single-VPN Sidecar

This image runs one UniVPN CLI connection inside a container. It is designed for
Kubernetes workloads that need a tunnel before the main container starts, and it
also works as a standalone privileged container for smoke tests.

This variant intentionally avoids desktop, X11, VNC, and dual-VPN behavior. The
goal is a small, predictable sidecar that is easy to stop cleanly.

## Key Behavior

- CLI-only runtime around `UniVPNCS`.
- Single VPN profile per container.
- Autogenerated profile support from `VPN_SERVER`, `VPN_PORT`, and
  `VPN_PROFILE_NAME`.
- Optional fast path with a mounted profile, `VPN_CONFIG_PATH`, or
  `VPN_CONFIG_B64`.
- Health checks based on the UniVPN process and tunnel route/interface state.
- Graceful termination: `connect.exp` traps `SIGTERM`/`SIGINT`, sends `q` to
  `UniVPNCS`, waits briefly, and lets the entrypoint force-kill only if needed.

## Third-Party Installer

The UniVPN Linux installer is not included. Place the vendor-provided installer
next to the Dockerfile before building:

```text
delivery/univpn-sidecar-single-vpn-minimal/univpn-linux-64-10781.18.1.0512.run
```

Do not commit the installer to Git. The repository `.gitignore` and local
`.dockerignore` exclude `*.run` by default.

## Environment Variables

Required for normal login:

- `VPN_SERVER`
- `VPN_PORT`
- `VPN_USER`
- `VPN_PASSWORD`
- `VPN_PROFILE_NAME`

Useful runtime tuning:

- `UNIVPN_CONNECT_TIMEOUT=120`
- `UNIVPN_MENU_SCAN_TIMEOUT=2`
- `UNIVPN_READY_POLL_INTERVAL=1`
- `UNIVPN_AUTOGENERATE_PROFILE=true`
- `UNIVPN_BOOTSTRAP_PROFILE=true`
- `UNIVPN_PROFILE_SETTLE_SECONDS=2`
- `UNIVPN_IMPORTED_PROFILE_SETTLE_SECONDS=0`
- `UNIVPN_DISCONNECT_TIMEOUT=15`
- `UNIVPN_DISCONNECT_WAIT=3`
- `UNIVPN_TUNNEL_IFACES=tun0`

Optional profile preload:

- `VPN_CONFIG_PATH=/usr/local/UniVPN/config/office.ini`
- `VPN_CONFIG_B64=<base64-encoded-profile>`

## Build

```bash
cd delivery/univpn-sidecar-single-vpn-minimal
./build.sh dev
```

The default image name is `univpn-sidecar`, so the command above builds:

```text
univpn-sidecar:dev
```

Override the base image or apt mirror if your network requires it:

```bash
BASE_IMAGE=ubuntu:22.04 APT_MIRROR=http://mirror.example.com/ubuntu ./build.sh dev
```

## Docker Smoke Test

```bash
cd delivery/univpn-sidecar-single-vpn-minimal
cp examples/.env.example .env
mkdir -p config logs

docker run -d \
  --name univpn-sidecar-test \
  --privileged \
  --cap-add NET_ADMIN \
  --cap-add NET_RAW \
  --cap-add SYS_MODULE \
  --device /dev/net/tun:/dev/net/tun \
  --env-file .env \
  -v "$PWD/config:/usr/local/UniVPN/config" \
  -v "$PWD/logs:/usr/local/UniVPN/log" \
  univpn-sidecar:dev
```

## Kubernetes

Two example manifests are included:

- `k8s-res-univpn.yaml`: standalone Deployment with a sample Secret.
- `k8s-sidecar-initcontainer-example.yaml`: initContainer/sidecar pattern for a
  workload that must wait for the VPN route before starting.

The container requires:

- privileged mode, or equivalent permissions for `/dev/net/tun`
- `NET_ADMIN`
- `NET_RAW`
- `SYS_MODULE` in environments where the tun device or modules need setup
- `/dev/net/tun` mounted from the host or available in the container runtime

Keep real VPN values in Kubernetes Secrets or your own secret manager. The
example manifests use placeholders only.

## Validation

```bash
bash -n build.sh entrypoint.sh healthcheck.sh
expect -n configure.exp
expect -n connect.exp
kubectl apply --dry-run=client -f k8s-res-univpn.yaml
kubectl apply --dry-run=client -f k8s-sidecar-initcontainer-example.yaml
```

`expect -n` may emit warnings depending on the local Expect version; syntax
errors should still return a non-zero exit code.

## Operational Notes

- This image is single-VPN by design. Use one container per VPN tunnel.
- Server-side auth and route delivery still determine most of the login time.
- If the VPN server keeps stale sessions when a container is killed, prefer a
  normal Kubernetes termination path so the `q` disconnect flow can run.
